# Architecture & State Management

LitLens AI strictly adheres to **Clean Architecture** principles. By completely separating the UI layer from the data and business logic layers, the application achieves high testability, maintainability, and extremely low latency during React rendering cycles.

## 🏗️ Layer Breakdown

1. **UI Layer (`app/`, `components/`)**
   - Pure, stateless presentation components.
   - React Server Components (RSC) handle initial data hydration.
   - Client Components bind exclusively to Custom Hooks for interactivity.
   - **Rule**: UI components *never* directly import Supabase clients or execute raw fetch queries.

2. **Custom Hooks (`hooks/`)**
   - The boundary between React lifecycle and application state.
   - Wraps Zustand stores and TanStack React Query mutations.
   - Enforces React 19 concurrent mode optimisations and `useMemo` for stable object references to prevent infinite re-renders.

3. **State Management (`stores/`)**
   - Global UI state is handled by Zustand.
   - Separated into logical domains: `useProjectStore`, `useChatStore`.
   - **Rule**: Stores only hold ephemeral UI state (e.g. `selectedProjectId`, `viewMode`, `searchQuery`). Server data is cached via TanStack Query.

4. **Services Layer (`services/`)**
   - Contains all business logic (e.g., `ProjectsService`, `PapersService`).
   - Responsible for orchestrating repository calls, transforming data payloads, and executing Server Actions.
   - **Rule**: Services are framework-agnostic. They should theoretically run seamlessly in a Node.js CLI script as well as a Next.js environment.

5. **Repositories Layer (`lib/repositories/`)**
   - Directly interfaces with external databases (Supabase PostgreSQL, Qdrant).
   - Responsible for raw SQL, RPC calls, and vector index queries.
   - Returns strictly typed interface objects (`Project`, `Paper`) to the Services layer, stripping away database-specific metadata (like raw PG row structures).

## 🗄️ Database Strategy

- **Relational Data**: Supabase (PostgreSQL). We utilize `select()` payloads tightly mapped to TypeScript interfaces, eliminating `SELECT *` overfetching.
- **Vector Data**: Qdrant Cloud. Handles all cosine-similarity vector embeddings generated by LangChain.

## 🔐 Security & Access Control

- **Authentication**: Clerk JWTs.
- **Database RLS**: Supabase Row-Level Security. Every API request from the frontend must include the Clerk JWT in the `Authorization: Bearer <token>` header. Supabase automatically verifies the JWT and applies RLS policies (e.g., `auth.uid() = user_id`) at the PostgreSQL engine level, ensuring zero data leakage.

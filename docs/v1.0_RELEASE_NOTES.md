# LitLens AI v1.0 Release Notes

We are thrilled to announce the official `v1.0` release of **LitLens AI**! 

This marks our transition from an experimental beta to an **Enterprise-grade AI research assistant** ready for serious production workloads.

## 🎉 What's New in v1.0

### 1. Robust Background Orchestration
We've migrated heavy PDF embedding and Large Language Model (LLM) processing from Vercel Serverless Functions to **Inngest**. This eliminates the 60-second timeout constraints, allowing researchers to upload massive multi-hundred-page documents without experiencing dropped requests or browser hangs.

### 2. Deep Caching Architecture
By implementing a tiered caching architecture (`RetrievalCache` and `PromptCache`) in our `AIOrchestrator`, we've drastically cut down latency and OpenAI API costs. Identical metadata extraction requests now resolve in `<50ms` rather than requiring redundant LLM inference passes.

### 3. Production Operations & Safety
- **Strict Environment Validation**: A Zod schema now validates the `.env` payload at server boot. Missing service keys will safely crash the container, preventing runtime exceptions.
- **Data Privacy via Redaction**: The new `StructuredLogger` intercepts all console logs, automatically sniffing and `[REDACTING]` sensitive information (like LLM keys and user prompts) before they ever reach DataDog/Vercel logs.
- **Global Error Boundaries**: Catastrophic UI crashes are now intercepted by a global Next.js boundary, ensuring the user gets a polished recovery screen instead of a raw stack trace.

### 4. Zero-Latency UI
We fully upgraded to **React 19** and optimized our frontend to use **Zustand** stores combined with **TanStack React Query**. This decouples UI state from server state, resulting in a buttery-smooth layout free from prop-drilling or `useEffect` infinite loop bugs.

## ⚠️ Upgrade Notes (for early access users)
If you are upgrading an existing instance:
1. Please pull the latest `.env.example`. You must now provide an `INNGEST_SIGNING_KEY`.
2. Do **not** run `supabase db reset`. There are no breaking schema changes in this version.
